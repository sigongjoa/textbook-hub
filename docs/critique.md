# 비판적 검토 (교육 현장 관점)

> 2026-04-20 자가검토. 초기 구상(README, usecases, blocks, processes)을 현장 선생님·SaaS 전략 관점에서 점검.

## 개념적 결함

### 1. "블록 단위 Fork"가 실무와 어긋남
선생님이 교재를 인식하는 단위는 **차시·단원**이지 블록이 아님. 블록이 좋아도 앞뒤 흐름(개념→예제→연습→심화)이 끊기면 학습 효과 나지 않음. "좋은 개념 블록 + 내 문제 블록 짜깁기"는 이상론. 실제 교재의 핵심 가치는 **시퀀싱**. GitHub 비유를 너무 직역한 감.

### 2. 공유 인센티브 공백
학원장·스타강사에게 좋은 블록 = 밥줄. 왜 공개함?
- GitHub는 커리어 자산·채용 레버리지가 있음 → 플랫폼 사용자는 여기 없음
- "사용됨 카운트 +1" 벨소리만으론 부족
- 공개 공유가 기본값인 UGC 플랫폼은 **경제 구조** 없이는 cold start 실패

### 3. 저작권 지뢰
한국 수학 콘텐츠의 상당수는 쎈·개념원리·올림포스·수특 파생. Fork 체인이 깊어지면 원저작권 추적 불가. 출판사 법무 리스크 높음 (EBS 수특 관련 판례 존재). "전면 공개" 기본값 자체가 위험.

## 기술 가정 점검

### 4. 한정 프로세스 = 함정
`problem.difficulty(delta=-1)` 같은 연산의 결과는 미묘하게 교실에 안 맞음. 숙련 선생님은 한 문장의 조사, 예시 하나의 문맥에 민감. 한정 프로세스는 **자유편집만큼 빠르지 않고, 결과는 미묘하게 어긋나는 중간지대**에 빠질 위험. math-verify는 정답성만 보고 **교육학적 타당성**(학년별 학습곡선 적합도)은 못 봄.

### 5. AI 역할 미정 = 핵심 미정
README에 "AI 역할 미정"이라고 쓴 순간 제품이 없는 것. 한정 프로세스 실행 엔진 = AI 품질 = 제품 품질. 미룰 수 없는 결정.

## 시장 포지셔닝

### 6. 기존 솔루션 대비 차별화 부재
iScream, 매쓰홀릭, 수학사랑 카페, 수만휘 자료실 등이 이미 공유 생태계 형성. **블록·Fork가 여기보다 나은 이유**가 초기 문서에 명시돼 있지 않음. "버전 관리 있음"만으로는 약함.

## 유효한 지점 (살릴 것)

- **UC-07 (학생별 Fork)**: 과외·소형학원 1:1 개별화 수요 강력.
- **정기고사 변형 생성** (수치 교체, 유사문제 N개): 즉시 수요.
- **`applied_ops` 계보 기록**: 교육 R&D 관점 가치 있음 (예: "난이도↓ 변형이 가장 인기" 같은 메타 통계).
